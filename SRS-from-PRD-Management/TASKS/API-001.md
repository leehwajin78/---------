## ✅ Task Breakdown (실행 계획)
- [ ] Zod 또는 TypeScript 인터페이스로 **Request DTO** 정의:
  ```typescript
  // DiagnoseRequest
  {
    answers: {
      q1: string;  // 객관식 선택지 (A|B|C|D|E)
      q2: string;
      q3: string;
      q4: string;
      q5: string;  // 단답형 가능 (REQ-FUNC-012: 객관식 90% + 단답형 10%)
    };
    leadInfo: {
      name: string;       // 필수
      phone: string;      // 필수
      email: string;      // 필수, 이메일 형식
      title?: string;     // 선택
      company?: string;   // 선택
      channel: string;    // 필수 (organic|referral|ad 등)
    };
  }
  ```
- [ ] Zod 또는 TypeScript 인터페이스로 **Response DTO** 정의:
  ```typescript
  // DiagnoseResponse (200 OK)
  {
    diagnosisId: string;    // UUID
    leadId: string;         // UUID
    brandScore: number;     // 0~100
    report: {
      summary: string;      // 종합 요약
      weaknesses: Array<{
        category: string;
        score: number;
        description: string;
        recommendation: string;
      }>;
      strengths: Array<{
        category: string;
        description: string;
      }>;
    };
    isInsincere: boolean;
    createdAt: string;      // ISO 8601
  }
  ```
- [ ] **에러 Response DTO** 정의:
  ```typescript
  // ErrorResponse
  {
    error: {
      code: string;        // 에러 코드
      message: string;     // 사용자 표시 메시지
      details?: object;    // 상세 정보 (필드별 검증 오류 등)
    };
  }
  ```
- [ ] **에러코드 목록** 정의:

  | HTTP Status | 에러코드 | 메시지 | 트리거 조건 |
  |---|---|---|---|
  | 400 | `INVALID_INPUT` | 입력 데이터 형식이 올바르지 않습니다 | 필수 필드 누락, 타입 불일치 |
  | 400 | `INVALID_EMAIL` | 올바른 이메일 형식이 아닙니다 | 이메일 정규식 불일치 |
  | 422 | `INSINCERE_RESPONSE` | 응답 패턴 이상 — 다시 한번 확인해주세요 | 5문항 전체 동일 선택 (REQ-FUNC-009) |
  | 429 | `RATE_LIMITED` | 요청이 너무 많습니다. 잠시 후 다시 시도해주세요 | 50 req/hr per IP 초과 |
  | 500 | `AI_SERVICE_ERROR` | 현재 AI 분석 서비스가 일시적으로 지연되고 있습니다 | Gemini API 장애 (REQ-FUNC-030) |
  | 503 | `SERVICE_UNAVAILABLE` | 서비스 점검 중입니다 | 시스템 점검 |

- [ ] Zod 스키마로 Request 유효성 검증 룰 구현 (이메일 regex, 필수 필드, answers 5문항 완성 여부)
- [ ] DTO 파일을 공유 모듈(`src/types/` 또는 `src/lib/dto/`)에 배치하여 프론트/백 공유
- [ ] 각 DTO에 JSDoc 주석으로 SRS 매핑 정보 기록

## 🧪 Acceptance Criteria (BDD/GWT)

**Scenario 1: 정상 Request 유효성 통과**
- Given: 5문항 응답(q1~q5)과 필수 리드 정보(name, phone, email, channel)가 모두 유효한 상태
- When: DiagnoseRequest Zod 스키마로 파싱하면
- Then: 에러 없이 파싱 성공하고, 타입 안전한 객체가 반환된다.

**Scenario 2: 필수 필드 누락 시 검증 실패**
- Given: `email` 필드가 누락된 Request가 주어진 상태
- When: Zod 스키마로 파싱하면
- Then: `INVALID_INPUT` 에러코드와 누락 필드 정보가 포함된 ZodError가 발생한다.

**Scenario 3: 이메일 형식 오류 검증**
- Given: `email`이 `"invalid-email"`인 Request가 주어진 상태
- When: Zod 스키마로 파싱하면
- Then: `INVALID_EMAIL` 에러코드와 "올바른 이메일 형식이 아닙니다" 메시지가 반환된다.

**Scenario 4: Response DTO 타입 안전성**
- Given: Gemini API가 브랜드 지수 75점과 약점 3건을 반환한 상태
- When: DiagnoseResponse DTO로 직렬화하면
- Then: `brandScore`가 number, `report.weaknesses`가 배열로 타입이 보장되고, 누락 필드가 없다.

## ⚙️ Technical & Non-Functional Constraints
- Rate Limit: 50 req/hr per IP (§6.1 API-01)
- 인증: 없음 (Public 엔드포인트)
- `answers` 필드 구조는 DIAGNOSIS 테이블의 JSONB `answers` 필드와 1:1 매핑
- Zod 스키마와 TypeScript 타입을 동시에 export하여 런타임/컴파일타임 양쪽 검증 보장
- Response의 `report` 구조는 Gemini API 프롬프트 출력 포맷에 의존 — 프롬프트 확정 후 세부 조정 가능
