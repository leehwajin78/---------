## 🎯 Summary
- 기능명: [DB-002] LEAD 테이블 스키마 및 마이그레이션 작성
- 목적: 잠재 고객(리드)의 정보를 저장·관리하기 위한 `leads` 테이블을 Prisma 스키마로 선언하고, 마이그레이션을 실행하여 데이터베이스에 반영한다. 리드 퍼널(진단→상담→전환)의 핵심 데이터 저장소이다.

## ✅ Task Breakdown (실행 계획)
- [ ] `schema.prisma`에 `LeadStatus` enum 정의: `new`, `contacted`, `converted`, `lost`
- [ ] `schema.prisma`에 `Lead` 모델 정의:
  - `id` — UUID, `@id @default(uuid())`
  - `name` — String, `@db.VarChar(100)`, NOT NULL
  - `phone` — String, `@db.VarChar(20)`, NOT NULL
  - `email` — String, `@db.VarChar(255)`, NOT NULL, `@unique`
  - `title` — String?, `@db.VarChar(200)`, NULLABLE
  - `company` — String?, `@db.VarChar(200)`, NULLABLE
  - `channel` — String, `@db.VarChar(50)`, NOT NULL (organic, referral, ad 등)
  - `status` — LeadStatus, `@default(new)`, NOT NULL
  - `createdAt` — DateTime, `@default(now())`
  - `updatedAt` — DateTime, `@updatedAt`
- [ ] Lead 모델에 관계(Relation) 정의:
  - `diagnoses` — Diagnosis[] (1:N)
  - `client` — Client? (1:0..1)
- [ ] `npx prisma migrate dev --name create-lead-table` 실행 — 마이그레이션 생성 및 적용
- [ ] `npx prisma generate` 실행 — Prisma Client 타입 갱신 확인
- [ ] SQLite 환경에서 enum 처리 호환성 검증 (SQLite는 native enum 미지원 → Prisma가 String으로 폴백하는지 확인)

## 🧪 Acceptance Criteria (BDD/GWT)

**Scenario 1: LEAD 테이블 생성 확인**
- Given: `schema.prisma`에 Lead 모델이 정의된 상태
- When: `npx prisma migrate dev`를 실행하면
- Then: 데이터베이스에 `leads` 테이블이 생성되고, 모든 필드(id, name, phone, email, title, company, channel, status, created_at, updated_at)가 SRS §6.2.1 명세와 일치한다.

**Scenario 2: email UNIQUE 제약 검증**
- Given: `leads` 테이블에 `test@example.com` 이메일의 레코드가 존재하는 상태
- When: 동일 이메일(`test@example.com`)로 새 레코드를 INSERT 시도하면
- Then: Prisma가 Unique constraint violation 에러를 발생시키고, 중복 레코드 삽입이 차단된다.

**Scenario 3: status 기본값 검증**
- Given: Lead 모델의 status 필드에 `@default(new)`가 설정된 상태
- When: status 값을 명시하지 않고 새 리드를 생성하면
- Then: 생성된 레코드의 status가 자동으로 `new`로 설정된다.

**Scenario 4: updatedAt 자동 갱신 검증**
- Given: 기존 리드 레코드가 존재하는 상태
- When: 해당 레코드의 status를 `contacted`로 업데이트하면
- Then: `updatedAt` 필드가 현재 시간으로 자동 갱신된다.

## ⚙️ Technical & Non-Functional Constraints
- **CON-08**: SQLite(로컬)에서 enum이 String으로 처리되는 동작을 반드시 검증할 것
- `email` 필드는 UNIQUE 제약 — 리드 중복 방지 (REQ-FUNC-020의 이메일 유효성 검증 기반)
- Prisma 명명 규칙: 모델명은 PascalCase(`Lead`), 필드명은 camelCase, DB 컬럼은 `@map`으로 snake_case 매핑
- `channel` 필드는 추후 확장을 위해 enum 대신 String으로 설계 (SRS 명세 준수)
