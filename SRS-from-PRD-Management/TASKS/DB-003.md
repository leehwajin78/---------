## ✅ Task Breakdown (실행 계획)
- [ ] `schema.prisma`에 `Diagnosis` 모델 정의:
  - `id` — UUID, `@id @default(uuid())`
  - `leadId` — UUID, FK → Lead.id, NOT NULL
  - `answers` — Json (PostgreSQL: JSONB, SQLite: String/Json) — 5문항 응답 원본
  - `report` — Json (PostgreSQL: JSONB, SQLite: String/Json) — AI 생성 리포트 (브랜드 지수, 약점 분석)
  - `brandScore` — Float, NOT NULL — 0~100점 브랜드 지수
  - `isInsincere` — Boolean, `@default(false)` — 불성실 응답 여부
  - `createdAt` — DateTime, `@default(now())`
- [ ] Lead ↔ Diagnosis 관계 설정: `lead Lead @relation(fields: [leadId], references: [id])`
- [ ] `brandScore` 필드에 대한 체크 제약(0~100) 처리 전략 결정:
  - Prisma는 CHECK 제약을 직접 지원하지 않으므로, 애플리케이션 레벨 검증 또는 `prisma migrate` 후 raw SQL로 CHECK 추가
- [ ] JSONB ↔ Json 호환성 테스트:
  - SQLite에서 Json 타입 필드에 객체를 저장/조회하는 동작 검증
  - PostgreSQL에서 JSONB로 정상 저장/쿼리되는지 검증
- [ ] `npx prisma migrate dev --name create-diagnosis-table` 실행
- [ ] `npx prisma generate` 실행 — 타입 갱신 확인

## 🧪 Acceptance Criteria (BDD/GWT)

**Scenario 1: DIAGNOSIS 테이블 생성 확인**
- Given: `schema.prisma`에 Diagnosis 모델이 정의된 상태
- When: `npx prisma migrate dev`를 실행하면
- Then: `diagnoses` 테이블이 생성되고, SRS §6.2.2의 모든 필드(id, lead_id, answers, report, brand_score, is_insincere, created_at)가 존재한다.

**Scenario 2: JSONB(answers) 필드 저장/조회 검증**
- Given: 5문항 응답 데이터 `{"q1": "A", "q2": "B", "q3": "C", "q4": "D", "q5": "A"}`가 주어진 상태
- When: 해당 데이터를 `answers` 필드에 저장하고 다시 조회하면
- Then: 저장된 JSON 객체가 원본과 동일하게 반환되며, 각 키-값 쌍이 보존된다.

**Scenario 3: FK(lead_id) 참조 무결성 검증**
- Given: `leads` 테이블에 존재하지 않는 UUID가 주어진 상태
- When: 해당 UUID를 `leadId`로 하여 Diagnosis 레코드를 생성 시도하면
- Then: Prisma가 Foreign key constraint violation 에러를 발생시키고, 레코드 생성이 차단된다.

**Scenario 4: brandScore 범위 검증 (애플리케이션 레벨)**
- Given: `brandScore`에 -10 또는 150이 입력된 상태
- When: 레코드 생성을 시도하면
- Then: 애플리케이션 레벨 검증에서 차단되고, "브랜드 지수는 0~100 범위여야 합니다" 에러가 반환된다.

**Scenario 5: isInsincere 기본값 검증**
- Given: `isInsincere` 값을 명시하지 않은 상태
- When: 새 Diagnosis 레코드를 생성하면
- Then: `isInsincere`가 자동으로 `false`로 설정된다.

## ⚙️ Technical & Non-Functional Constraints
- **CON-08 (핵심)**: `answers`와 `report` 필드는 JSONB(PostgreSQL) / Json(SQLite) 호환성을 반드시 검증할 것
- `brandScore`의 CHECK(0~100) 제약은 DB 수준에서는 raw SQL migration으로, 애플리케이션에서는 Zod 등 검증 라이브러리로 이중 방어
- Diagnosis는 `updatedAt`이 없음 (SRS 명세상 생성 후 수정 불가 — 불변 레코드)
- 한 Lead에 여러 Diagnosis가 가능 (재진단 시나리오)
